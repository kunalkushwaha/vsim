The one-liner

“Remotion for real 3D.”

A code-first, deterministic 3D engine that exports short videos and embeds live interactive scenes from the same project — with real simulation and an AI copilot. Open-core, developers first.

A pitch a developer and an investor both understand in one sentence.

Why now · the gap

Everyone proved a piece. Nobody owns the middle.

Remotion

Proved: code → video is a category.
Misses: 2D React-DOM only. No real 3D or simulation.

Spline / Theatre.js

Proved: web 3D editing + timelines.
Misses: weak sim, video export, AI, data scale.

Blender

Proved: full 3D power.
Misses: desktop-only, steep, no web/embed/scale.

Runway / AI video

Proved: text → video is magic.
Misses: no scene graph, no determinism, no precision.

The wedge: controllable, reproducible, code-defined 3D that ships as both video and live embed.

The three core bets

What makes it hard is what makes it defensible

01

One canonical document

Timeline, code, and AI all read/write the same serializable scene. Must round-trip. The moat — and the hardest UX problem (the Webflow↔code problem).

02

Determinism is the game

Video + interactive + N personalized variants + simulation ⇒ fixed timestep, seeded RNG. preview == final render == every variant. Rapier is deterministic by design.

03

Same engine, two modes

Real-time loop for the embed; deterministic frame-stepping for export. One codebase — no second renderer, no “render surprise.”

Stack decision

Three.js + Rapier now — WebGPU core later

v1 runtime

Three.js + Rapier

Largest dev community + React-Three-Fiber (perfect for code authoring). Rapier is Rust/WASM and deterministic by design. Fastest path to a working SDK for your first wedge: developers.

the future-proofing

Thin engine interface

Keep the scene document & public API independent of the renderer. Later swap in a custom WebGPU compute core for GPU particles/fluids — without breaking user scenes.

Babylon.js = strong runner-up (more batteries) but loses on the dev-ecosystem dimension that is the wedge. Audio is core: map beats → frame numbers so audio-reactive motion stays reproducible.

MVP · the magic moment

Code → Video. The developer “aha.”

Write a small 3D scene in code, with physics. Hit render. Get an MP4 with real 3D + deterministic simulation.

Top risks · open questions

What can sink it

Architecture

Code ↔ timeline round-trip

The hardest UX problem. Decide the source-of-truth model early — don’t naively two-way sync.

Cost / COGS

Headless GPU rendering

This is both the cost center and the monetization hook. Validate the render pipeline early.

Determinism

Cross-platform floats

Mitigated by running finals through the same pipeline; still verify Rapier reproducibility.

Discipline

Scope creep

4 audiences × 4 use cases × 4 sim types = a platform, not an MVP. Hold at “code → video.”

Next step

Build the code → video demo.
Everything else follows.

One canonical document. One deterministic runtime. Two outputs. Win developers first, then layer the editor, AI, and templates on top of an engine that already works.
